# 工作流基础框架 - 需求文档

## 项目介绍

本项目旨在为智慧图书馆小程序构建一个通用、可扩展的工作流基础框架,支持从简单的单步审批(如用户注册)到复杂的多步骤流转(如购车流程)等多种业务场景。该框架需要支持条件分支、回退、补充资料、多部门协同审批等复杂功能,为后续所有工作流相关业务提供统一的技术底座。

## 核心需求

### 需求 1 - 工作流模板配置管理

**用户故事**: 作为系统管理员,我希望能通过配置工作流模板来定义不同业务类型的审批流程,这样我可以灵活调整流程而无需修改代码。

**验收标准**:
1. While 系统管理员已登录, when 管理员创建或修改工作流模板, the 系统应支持定义流程名称、版本号、步骤列表、步骤间的流转条件和审批人配置。
2. When 工作流模板被修改, the 系统应自动递增版本号并保留历史版本。
3. While 流程模板存在多个版本, when 工单被提交, the 系统应使用该模板的最新激活版本。
4. When 管理员禁用某个模板, the 系统应标记该模板为非激活状态,但不删除该模板及其关联的历史工单。
5. While 模板中的步骤包含审批人配置, when 配置审批人, the 系统应支持多种审批人类型(具体用户、角色、部门、动态表达式等)。

### 需求 2 - 工单提交与启动流程

**用户故事**: 作为普通用户,我希望能提交各类业务申请(如购车申请、请假申请等),系统自动根据业务类型匹配对应的工作流模板并启动审批流程。

**验收标准**:
1. While 用户已注册登录, when 用户提交业务申请并填写必要的业务数据, the 系统应验证业务数据的完整性和合法性。
2. When 业务数据验证通过, the 系统应查询对应的激活状态的工作流模板。
3. While 查询到工作流模板, when 系统评估模板中的步骤条件, the 系统应基于业务数据筛选出实际需要执行的步骤序列。
4. When 步骤序列确定, the 系统应创建工单记录并存储流程配置快照。
5. While 工单创建成功, when 流程启动, the 系统应自动创建第一个待审批任务节点并通知对应审批人。

### 需求 3 - 任务审批与流程流转

**用户故事**: 作为审批人,我希望能在待办列表中查看分配给我的任务,对任务进行通过或驳回操作,并系统自动将流程推进到下一步或结束流程。

**验收标准**:
1. While 审批人有待处理的任务, when 审批人打开任务详情, the 系统应显示工单的完整业务数据、当前步骤信息、前置审批历史和附件资料。
2. When 审批人执行通过操作, the 系统应验证审批人权限并更新当前任务状态为"已通过"。
3. While 当前任务被通过, when 存在下一步骤, the 系统应自动创建下一步的任务节点并通知下一步审批人。
4. When 审批人执行驳回操作, the 系统应记录驳回原因并根据配置决定是回退到上一节点、回退到申请人、还是直接结束流程。
5. While 流程到达最后一步且被通过, when 系统检测到无下一步骤, the 系统应将工单状态标记为"已完成"并执行流程完成后的业务逻辑(如更新业务数据、发送通知等)。
6. When 任务超时未处理, the 系统应根据配置自动发送提醒通知或执行超时处理策略(如自动通过、自动驳回或转交上级)。

### 需求 4 - 流程回退与补充资料

**用户故事**: 作为审批人或申请人,我希望在流程中能够将工单回退到申请人补充资料,或回退到之前的审批节点重新审核,以处理资料不完整或审批有误的情况。

**验收标准**:
1. While 工单处于审批中状态, when 审批人选择"退回补充资料", the 系统应将工单状态标记为"待补充"并通知申请人。
2. When 申请人补充资料后重新提交, the 系统应将工单从"待补充"状态恢复为"审批中",并将流程定位到被退回的步骤。
3. While 审批人选择"退回上一步", when 存在上一个审批节点, the 系统应将任务状态更新为"已退回"并重新激活上一个任务节点。
4. When 流程被退回, the 系统应记录退回原因和退回路径以供审计追踪。
5. While 申请人被要求补充资料, when 申请人查看工单详情, the 系统应清晰显示需要补充的字段和退回原因说明。

### 需求 5 - 并行审批与会签机制

**用户故事**: 作为流程设计者,我希望某些步骤可以支持多个审批人并行审批,并根据策略决定是否需要所有审批人都通过(会签)或只需一人通过即可。

**验收标准**:
1. While 配置工作流模板, when 管理员将步骤类型设置为"并行审批", the 系统应允许为该步骤配置多个审批人。
2. When 并行步骤被启动, the 系统应为每个审批人创建独立的任务节点,这些任务可以同时被处理。
3. While 并行步骤设置为会签模式, when 系统判断并行任务的完成条件, the 系统应在所有审批人都通过后才将步骤标记为完成。
4. When 并行步骤设置为非会签模式(一人通过即可), the 系统应在任一审批人通过后立即将其他未处理的任务标记为"已取消"并推进流程。
5. While 并行步骤中存在审批人驳回, when 任一审批人驳回, the 系统应立即取消该步骤的其他未处理任务并根据配置决定是否回退或结束流程。

### 需求 6 - 条件分支与动态路由

**用户故事**: 作为流程设计者,我希望流程能根据业务数据的不同(如请假天数、购车金额等)自动选择不同的审批路径,以适应不同场景的审批要求。

**验收标准**:
1. While 配置工作流模板, when 管理员为步骤配置条件表达式, the 系统应支持基于业务数据字段的比较运算符(大于、小于、等于、包含等)。
2. When 工单启动并评估步骤条件, the 系统应计算每个条件的布尔结果并过滤出满足条件的步骤。
3. While 存在多个满足条件的步骤, when 这些步骤配置为互斥分支, the 系统应只保留第一个满足条件的步骤。
4. When 条件表达式引用业务数据中不存在的字段, the 系统应将该条件评估结果设为false。
5. While 流程执行过程中业务数据被修改, when 流程需要重新评估后续步骤, the 系统应基于最新的业务数据重新计算条件分支。

### 需求 7 - 流程监控与统计分析

**用户故事**: 作为管理员或部门负责人,我希望能够查看各工作流的执行情况、审批效率统计和异常报警,以便优化流程和监控风险。

**验收标准**:
1. While 管理员访问监控页面, when 系统展示流程统计概览, the 系统应显示各模板的工单总数、完成率、平均审批时长、超时次数等关键指标。
2. When 管理员筛选指定时间段或流程类型, the 系统应基于筛选条件实时更新统计图表和数据列表。
3. While 存在超时未处理的任务, when 系统检测到超时, the 系统应自动发送通知提醒相关人员并记录超时事件。
4. When 管理员查看流程详情, the 系统应以时间轴方式展示完整的审批路径、各节点的处理时长、审批意见和附件。
5. While 管理员导出统计数据, when 管理员选择导出格式, the 系统应生成Excel或CSV格式的统计报表。

### 需求 8 - 审批通知与消息推送

**用户故事**: 作为用户或审批人,我希望在工作流的关键节点(任务分配、审批结果、超时提醒等)能及时收到通知推送,确保不遗漏重要信息。

**验收标准**:
1. While 任务节点被创建或分配, when 系统执行通知逻辑, the 系统应通过微信订阅消息向审批人推送待办任务通知。
2. When 审批人执行审批操作, the 系统应向申请人推送审批结果通知(通过或驳回)并附带审批意见。
3. While 任务接近或超过超时时间, when 系统触发超时提醒, the 系统应向审批人发送催办通知。
4. When 流程回退到申请人补充资料, the 系统应向申请人推送待补充资料通知并明确补充要求。
5. While 用户订阅消息授权失效, when 系统发送通知失败, the 系统应记录失败日志并在应用内显示系统通知消息。

### 需求 9 - 流程历史与审计追踪

**用户故事**: 作为审计人员或管理员,我希望能够查询任何工单的完整操作历史,包括谁在什么时间做了什么操作,以满足合规审计要求。

**验收标准**:
1. While 工单状态发生变更, when 系统记录操作日志, the 系统应记录操作时间、操作人openid、操作类型、操作前后数据快照和操作描述。
2. When 审计人员查询工单历史, the 系统应返回按时间倒序排列的完整操作日志列表。
3. While 审计人员需要追溯某个操作, when 点击操作记录, the 系统应显示该操作的详细信息和数据快照对比。
4. When 系统执行自动操作(如超时处理), the 系统应记录操作类型为"系统自动执行"并标注触发原因。
5. While 工单被删除或归档, when 系统清理数据, the 系统应至少保留审计日志的时间满足合规要求(建议至少1年)。

### 需求 10 - 权限控制与数据安全

**用户故事**: 作为系统管理员,我希望工作流系统能够严格控制数据访问权限,确保用户只能查看和操作自己有权限的工单和任务。

**验收标准**:
1. While 用户访问工单列表, when 系统查询数据库, the 系统应根据用户角色和openid过滤出该用户有权限查看的工单(申请人只看自己的,审批人看分配给自己的)。
2. When 普通用户尝试访问他人的工单, the 系统应拒绝访问并返回权限错误。
3. While 审批人执行审批操作, when 系统验证审批权限, the 系统应检查该审批人是否是当前任务节点的指定审批人或其代理。
4. When 审批人查看业务数据, the 系统应根据数据敏感级别决定是否脱敏显示(如身份证号、手机号等)。
5. While 用户通过API直接调用工作流接口, when 系统执行权限校验, the 系统应验证请求者的openid和签名以防止未授权访问。

## 非功能性需求

### 性能要求
- 单次工单提交响应时间不超过2秒
- 工单列表查询响应时间不超过1秒
- 支持1000+并发用户同时使用工作流系统

### 可用性要求
- 工作流引擎可用性达到99.9%
- 关键审批通知延迟不超过30秒

### 可扩展性要求
- 支持新增业务类型时无需修改核心引擎代码
- 支持通过配置扩展审批人解析规则和条件判断逻辑

### 安全性要求
- 所有操作均需记录完整审计日志
- 敏感数据在传输和存储时加密
- 实现基于角色的访问控制(RBAC)
