## Phase 3: 工作流引擎核心功能

- [ ] 3.1 实现条件评估算法
  - 实现`evaluateSteps()`函数,基于业务数据筛选步骤
  - 实现`evaluateCondition()`函数,支持多种运算符(gt/lt/eq/neq/gte/lte/in/contains)
  - 实现`getNestedValue()`函数,支持嵌套字段访问(如`businessData.leave.days`)
  - 编写单元测试覆盖各种条件场景
  - _需求: 6_

- [ ] 3.2 实现审批人解析逻辑
  - 实现`resolveApprovers()`函数,支持4种审批人类型
  - 实现`getUsersByRole()`函数,从用户表按角色查询
  - 实现`getUsersByDept()`函数,从用户表按部门查询
  - 实现`evaluateApproverExpression()`函数,支持动态表达式(如`applicant.leader`)
  - _需求: 1, 2, 3_

- [ ] 3.3 实现工单提交流程
  - 实现`startWorkflow()`函数:查询模板→评估条件→创建工单→创建首个任务
  - 实现`generateOrderNo()`函数,生成工单编号
  - 实现快照保存逻辑,复制模板配置到工单的workflowSnapshot
  - 编写日志记录逻辑,记录工单提交事件
  - 编写单元测试验证提交流程
  - _需求: 2_

- [ ] 3.4 实现任务创建逻辑
  - 实现`createTasks()`函数,根据步骤配置创建一个或多个任务
  - 实现`generateParallelGroupId()`函数,为并行步骤生成组ID
  - 支持串行步骤创建单个任务
  - 支持并行步骤创建多个任务,共享parallelGroupId
  - 设置任务的超时时间(timeoutAt)
  - _需求: 3, 5_

- [ ] 3.5 实现任务审批流程
  - 实现`approveTask()`函数:验证权限→更新任务→处理流转
  - 实现`hasPermission()`函数,验证审批人是否有权操作任务
  - 实现`handleApproval()`函数,处理审批通过逻辑
  - 实现`handleRejection()`函数,处理审批驳回逻辑
  - 实现`handleReturn()`函数,处理退回逻辑
  - 编写单元测试覆盖各种审批场景
  - _需求: 3_

- [ ] 3.6 实现并行审批逻辑
  - 实现`getParallelTasks()`函数,查询同组的所有并行任务
  - 在`handleApproval()`中判断并行步骤是否全部完成
  - 支持会签策略:所有任务都通过才推进
  - 支持或签策略:任一任务通过就取消其他任务
  - 实现取消未处理并行任务的逻辑
  - _需求: 5_

- [ ] 3.7 实现流程推进逻辑
  - 实现`checkNextStep()`函数,检查是否有下一步骤
  - 实现`findNextStep()`函数,查找满足条件的下一步骤
  - 支持动态评估下一步骤的条件(如果业务数据被修改)
  - 实现创建下一步任务的逻辑
  - 更新工单的currentStep字段
  - _需求: 3, 6_

- [ ] 3.8 实现流程完成逻辑
  - 实现`completeWorkflow()`函数:更新工单状态→记录日志→发送通知
  - 计算流程总耗时(totalDuration)
  - 执行流程完成后的业务逻辑钩子(可根据需求扩展)
  - 发送完成通知给申请人
  - 编写单元测试验证完成逻辑
  - _需求: 3_

## Phase 4: 回退与补充资料功能

- [ ] 4.1 实现退回到申请人逻辑
  - 在`handleReturn()`中实现退回到申请人的逻辑
  - 将工单状态更新为`supplement`(待补充)
  - 设置`needSupplement=true`和`supplementReason`
  - 取消当前步骤和后续步骤的所有未处理任务
  - 发送退回通知给申请人
  - _需求: 4_

- [ ] 4.2 实现退回到上一步逻辑
  - 在`handleReturn()`中实现退回到上一个步骤的逻辑
  - 将被退回的任务状态更新为`returned`
  - 重新激活上一个步骤的任务(状态从`approved`改为`pending`)
  - 记录退回路径(returnedFromStep, returnReason)
  - 发送退回通知给上一个审批人
  - _需求: 4_

- [ ] 4.3 实现补充资料功能
  - 实现`supplementOrder()`函数:允许申请人补充业务数据
  - 验证补充的数据格式和必填字段
  - 将补充数据合并到`businessData`
  - 增加`supplementCount`计数
  - 将工单状态恢复为`pending`
  - 恢复被退回步骤的任务状态为`pending`
  - 记录补充操作日志
  - _需求: 4_

- [ ] 4.4 实现重新提交流程
  - 实现`resubmitOrder()`函数:申请人补充资料后重新提交
  - 验证工单是否处于`supplement`状态
  - 创建新的任务节点(或恢复之前的任务)
  - 发送通知给审批人
  - _需求: 4_

## Phase 5: 超时处理机制

- [ ] 5.1 配置云函数定时触发器
  - 为`workflowEngine`云函数配置定时触发器
  - 设置触发规则为每5分钟执行一次超时检查
  - _需求: 3_

- [ ] 5.2 实现超时检测逻辑
  - 实现`handleTimeoutTasks()`函数
  - 查询所有`taskStatus=pending`且`timeoutAt<=当前时间`的任务
  - 批量处理超时任务
  - _需求: 3_

- [ ] 5.3 实现超时处理策略
  - 支持`auto_approve`:自动通过,调用`approveTask(taskId, 'approve', '系统自动通过', 'system')`
  - 支持`auto_reject`:自动驳回,调用`approveTask(taskId, 'reject', '系统超时自动驳回', 'system')`
  - 支持`escalate`:升级到上级,创建新的审批任务
  - 支持`remind`:发送提醒通知,调用通知服务
  - 将任务标记为`isTimeout=true`
  - _需求: 3, 6_

- [ ] 5.4 实现超时通知功能
  - 实现`sendTimeoutNotification()`函数
  - 发送订阅消息提醒审批人
  - 在应用内显示系统通知
  - 记录超时事件日志
  - _需求: 8_
